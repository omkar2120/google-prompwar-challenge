# Architecture

MonsoonMitra is a React 18 + Vite PWA. This document explains the module layout
and how data flows through the app. For feature-level detail see the
[README](../README.md); for the test strategy see [TESTING.md](../TESTING.md).

## Layered design

```
UI (routes/, components/)
        │  calls
        ▼
Hooks (hooks/)  ──uses──►  State (store/)   ← client state (profile, theme, alerts)
        │
        │  calls
        ▼
Lib (lib/)  ──►  External APIs (Open-Meteo, Groq, Firebase)
```

- **`routes/`** — one file per screen (Home, Onboarding, Dashboard, Plan,
  Checklist, Travel, Community, Recovery, Login, Profile). Screens compose
  hooks + presentational components; they hold only local UI state.
- **`components/`** — presentational + small stateful widgets, grouped by domain
  (`ui/`, `weather/`, `checklist/`, `chat/`, `map/`). Every component with props
  declares `PropTypes`.
- **`hooks/`** — the behavioral layer. Data-fetching hooks wrap React Query
  (`useWeather`, `useGeocode`, `useWeatherInterpretation`, `useCommunitySummary`,
  `usePreparednessPlan`); stateful feature hooks own side effects
  (`useAlertsEngine`, `useGroqChat`, `useVoiceRecorder`, `useAuth`,
  `useCommunityReports`, `useOnlineStatus`).
- **`lib/`** — framework-agnostic, side-effect-light modules and the only place
  that talks to the network:
  - `openMeteo.js` — weather/geocoding fetch + deterministic severity scoring.
  - `groqClient.js` — every Groq request (chat, JSON, Whisper) with one shared
    error path.
  - `prompts/` — one system-prompt builder per AI feature (language interpolated).
  - `validation.js` — input sanitization + size caps for all user input.
  - `exportUtils.js` — PDF export + Web Share / WhatsApp.
  - `datetime.js` — shared relative-time formatting.
  - `firebase.js` — optional Auth + Firestore, with a guest/local fallback.
- **`store/appStore.js`** — Zustand store for *client* state only (household
  profile, language, theme, alert history, checklist progress, activity log).
  Server state deliberately lives in React Query, not here.
- **`i18n/`** — react-i18next setup + en/hi/mr/ta translations.
- **`types/index.js`** — JSDoc `@typedef` data contracts (checked via `checkJs`).

## Core data flow: from weather to advice

1. **Location** comes from the household profile (`store`) or a `LocationPicker`
   (geolocation + Open-Meteo geocoding in `lib/openMeteo.js`).
2. **Forecast** is fetched by `useWeather` → `fetchForecast`, cached by React
   Query.
3. **Severity** is computed *deterministically* by `assessSeverity`
   (`classifyRain` + wind/probability escalation) against IMD thresholds — the
   LLM never decides severity.
4. **AI phrasing/plans** are generated by hooks calling `groqClient` with a
   builder from `prompts/`; the selected language is interpolated so output is
   native. Empty/failed AI responses fall back to deterministic text (e.g. the
   alerts engine never fires a blank alert).
5. **Persistence**: profile, alert history, and checklist progress persist to
   `localStorage` via the store; the service worker (`sw.js`) caches the last
   forecast/plan for offline viewing.

## Design principles

- **Deterministic vs. generative split** — thresholds decide *what* is risky;
  the LLM only explains and phrases.
- **No fake data** — React Query surfaces real loading/error/empty states; on
  failure the UI offers retry rather than substituting stale values silently.
- **Graceful degradation** — the app runs with no Groq key (clear prompts to add
  one) and no Firebase (device-local community mode).
- **Single responsibility** — network access is confined to `lib/`; components
  stay presentational; the store holds only client state.
